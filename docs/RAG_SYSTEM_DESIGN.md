# RAG 系统功能说明文档

## 📌 概述

本文档详细介绍当前项目实现的 RAG（Retrieval-Augmented Generation）知识库系统的核心功能模块、技术实现原理和使用方法。

   ▼

## 🔧 核心功能模块

### 1. 混合检索（Hybrid Search）

#### 功能说明
结合向量检索（语义相似度）和 BM25 关键词检索（字面匹配）的优势，提升检索的全面性和准确性。

#### 技术实现

**整体流程**：
1. **并行检索**：同时对同一个查询执行向量检索和 BM25 检索
2. **独立评分**：两种检索方式各自产生带评分的结果集
3. **结果融合**：使用 RRF 算法合并两个结果集
4. **统一排序**：按融合后的分数重新排序

**向量检索**：
- 使用 Milvus 向量数据库存储文档片段的向量表示
- 查询时将用户问题转换为向量（使用相同的嵌入模型）
- 执行余弦相似度搜索，返回最相似的 Top-K 个文档
- 每个结果包含：文档 ID、内容、向量距离分数、元数据

**BM25 检索**：
- 基于 Lucene 构建倒排索引
- 对文档内容进行分词处理（支持中文分词）
- 计算词频（TF）和逆文档频率（IDF）
- 使用 BM25 公式计算查询与文档的相关性分数
- 考虑文档长度归一化，避免长文档占优

**关键设计**：
- 两种检索方式独立执行，互不干扰
- 每个查询扩大检索范围（Top-K × 3），为融合提供充足候选
- 支持命名空间过滤，确保在指定知识范围内检索

**实现类**：
- `HybridRetrievalService` - 混合检索服务
- `BM25Retriever` - BM25 检索器
- `SparseRetriever` - 稀疏检索接口

### 2. RRF 分数融合（Reciprocal Rank Fusion）

#### 功能说明
将多个检索结果集（向量检索结果 + BM25 检索结果）合并为一个统一排序的结果集，无需关心各检索方式的原始分数。

#### 技术实现

**核心算法**：
```
融合分数 = Σ (1 / (k + 排名))
```
其中 k 是平滑参数（通常为 60），排名是文档在某个结果集中的位置（从 1 开始）

**执行步骤**：
1. **收集结果**：从向量检索和 BM25 检索获取两个结果集
2. **去重合并**：使用文档 ID 作为唯一标识，合并所有文档
3. **计算 RRF 分数**：
   - 对于每个文档，查找它在各个结果集中的排名
   - 如果文档在某个结果集中不存在，该结果集的贡献为 0
   - 累加所有结果集的倒数排名分数
4. **排序返回**：按 RRF 分数降序排列，取 Top-K

**优势**：
- 无需考虑原始分数的量纲差异（向量距离 vs BM25 分数）
- 排名比绝对分数更稳定
- 平滑参数 k 控制排名的影响力，避免单一检索方式主导

**参数调优**：
- k 值越大，排名的影响力越小
- k 值越小，高排名文档的优势越明显
- 默认 k=60 是经验值，适合大多数场景

**实现类**：
- `RRFFusion` - RRF 融合器
- `ScoreFusion` - 分数融合接口

### 3. 语义去重（Semantic Deduplication）

#### 功能说明
识别并移除检索结果中语义重复的文档片段，避免返回大量相似内容，提升结果多样性。

#### 技术实现

**去重流程**：
1. **向量化表示**：使用嵌入模型将每个文档片段转换为向量
2. **两两比较**：计算所有文档对之间的余弦相似度
3. **阈值过滤**：相似度超过阈值（如 0.85）的文档对视为重复
4. **保留最优**：对于重复的文档对，保留检索分数更高的那个
5. **迭代处理**：重复上述过程直到没有新的重复对

**相似度计算**：
- 使用余弦相似度：cos(θ) = (A·B) / (||A|| × ||B||)
- 值域 [-1, 1]，越接近 1 表示越相似
- 阈值 0.85 是经验值，可根据实际需求调整

**优化策略**：
- 对于小规模结果集（<50 条），直接两两计算
- 对于大规模结果集，可先用局部敏感哈希（LSH）预筛选
- 缓存已计算的相似度，避免重复计算

**边界处理**：
- 部分重叠的文档（如父子索引）需要特殊处理
- 考虑使用 Jaccard 相似度作为补充判断
- 保留 headingPath 信息，优先保留结构更完整的文档

**实现类**：
- `SemanticDeduplicator` - 语义去重器

### 4. MMR 重排序（Maximal Marginal Relevance）

#### 功能说明
在保持相关性的同时最大化结果多样性，避免返回的内容过于集中在某个主题。

#### 技术实现

**核心思想**：
迭代选择文档，每次选择既与查询相关又与已选文档差异最大的文档。

**MMR 公式**：
```
MMR = λ × 相关性 (文档，查询) - (1-λ) × 最大相似度 (文档，已选文档)
```

**执行步骤**：
1. **初始化**：选择与查询最相关的文档作为第一个结果
2. **迭代选择**：
   - 对于剩余候选文档，计算 MMR 分数
   - 相关性：文档与查询的余弦相似度
   - 相似度：候选文档与所有已选文档的最大相似度
   - 选择 MMR 分数最高的文档加入结果集
3. **终止条件**：达到目标数量（Top-K）或没有候选文档

**参数调优**：
- λ = 1.0：完全基于相关性（退化为普通排序）
- λ = 0.5：相关性和多样性各占一半
- λ = 0.7：偏重相关性，兼顾多样性（推荐值）
- λ = 0.3：偏重多样性，适合探索性查询

**应用场景**：
- 用户查询意图不明确时
- 需要覆盖多个子主题时
- 避免信息冗余时

**实现类**：
- `MMRReranker` - MMR 重排序器

### 5. 查询重写（Query Rewriting）

#### 功能说明
将用户的自然语言查询转换为更适合检索的形式，提升检索准确性。

#### 技术实现

**重写策略**：
1. **语法规范化**：
   - 纠正拼写错误
   - 统一术语表述
   - 补充省略成分

2. **语义优化**：
   - 将口语化表达转为专业术语
   - 将复杂问句简化为核心查询
   - 去除无关词汇和语气词

3. **上下文补全**：
   - 在多轮对话中，使用代词消解补充完整信息
   - 结合对话历史理解当前查询的真实意图

**实现方式**：
- 使用 LLM 进行端到端的重写
- Prompt 设计：「请将以下查询改写为更适合检索的形式：{query}」
- 提供示例 Few-shot 学习
- 约束输出格式，确保只返回改写后的查询

**典型转换**：
- 「我想了解一下那个什么 AI 代理的东西」→ 「AI Agent 介绍」
- 「怎么弄那个文件上传啊？」→ 「文件上传方法」
- 「它有什么特点？」（上下文中「它」指 AI Agent）→ 「AI Agent 有什么特点」

**实现类**：
- `LLMQueryRewriter` - LLM 查询重写器
- `QueryRewriter` - 查询重写接口

### 6. HyDE 假设文档嵌入（Hypothetical Document Embeddings）

#### 功能说明
先生成一个假设的文档来回答用户问题，然后用这个假设文档的向量进行检索，找到真正相关的文档。

#### 技术实现

**核心洞察**：
- 用户查询通常很短，与文档的语义空间不匹配
- 假设文档更接近真实文档的表述方式
- 用假设文档检索可以跨越词汇不匹配的鸿沟

**执行流程**：
1. **生成假设文档**：
   - 使用 LLM 根据问题生成一个假设性答案
   - Prompt：「请根据问题生成一个假设的文档来回答这个问题：{query}」
   - 控制生成长度（200-500 字），避免过长

2. **向量化**：
   - 将假设文档转换为向量（使用与文档库相同的嵌入模型）

3. **检索**：
   - 用假设文档的向量在 Milvus 中进行相似度搜索
   - 返回与假设文档最相似的真实文档

4. **融合**：
   - 将假设文档检索的结果与原始查询检索的结果合并
   - 使用 RRF 或其他融合策略

**优势**：
- 解决查询与文档长度不匹配问题
- 跨越词汇鸿沟（查询用词≠文档用词）
- 特别适合开放域问答和复杂查询

**注意事项**：
- 假设文档可能包含错误信息，仅用于检索，不作为答案
- 需要标注来源，告知用户哪些是假设内容
- 控制生成质量，避免幻觉影响检索

**实现类**：
- `HydeGenerator` - HyDE 生成器

### 7. MQE 多查询扩展（Multiple Query Expansion）

#### 功能说明
基于原始查询生成多个语义相似但表述不同的查询，扩大检索覆盖范围，提升召回率。

#### 技术实现

**扩展策略**：
1. **同义词替换**：
   - 使用词典或词向量找到关键词的同义词
   - 例如：「AI Agent」→ 「智能代理」、「AI 助手」、「人工智能代理」

2. **句式变换**：
   - 将陈述句改为疑问句
   - 主动语态改为被动语态
   - 例如：「介绍 AI Agent」→ 「AI Agent 是什么」「请说明 AI Agent 的定义」

3. **角度扩展**：
   - 从不同维度提问
   - 例如查询「AI Agent」扩展为：
     - 「AI Agent 的定义」
     - 「AI Agent 的特点」
     - 「AI Agent 的应用场景」
     - 「AI Agent 如何实现」

**实现方式**：
- 使用 LLM 批量生成：「请生成 5 个与以下问题语义相似但表述不同的问题：{query}」
- 解析 LLM 输出，提取多个查询
- 去重处理，移除语义重复的查询

**检索融合**：
- 对每个扩展查询执行检索
- 合并所有结果，去重（基于文档 ID）
- 使用 RRF 融合各查询的结果
- 每个查询的 Top-K 适当缩小，避免总结果过多

**优势**：
- 覆盖用户可能忽略的相关文档
- 降低单一查询的偶然性
- 特别适合专业术语和复杂概念

**实现类**：
- `QueryExpander` - 查询扩展器
- `DictionarySynonymExpander` - 词典同义词扩展器

### 8. Milvus 向量相似度搜索

#### 功能说明
使用 Milvus 向量数据库高效存储和检索文档的向量表示，实现基于语义的相似度搜索。

#### 技术实现

**向量存储**：
1. **文档向量化**：
   - 使用嵌入模型（如 text-embedding-ada-002、bge-large-zh）将文档片段转换为固定维度的向量
   - 常用维度：768、1536、2048
   - 向量归一化处理，便于余弦相似度计算

2. **Milvus 集合设计**：
   - 主键：文档 ID（UUID）
   - 向量字段：存储嵌入向量
   - 标量字段：content（内容）、metadata（JSON）、ragNamespace（命名空间）
   - 索引类型：HNSW（高效导航小世界）

3. **索引构建**：
   - HNSW 参数：M=16（邻居数），efConstruction=200（构建精度）
   - 平衡索引构建时间和检索速度
   - 支持增量更新（新文档插入）

**相似度搜索**：
1. **查询向量化**：使用相同嵌入模型将查询转为向量
2. **近似最近邻搜索**：
   - 使用 HNSW 算法在向量空间中快速定位最近邻
   - ef 参数：搜索时的探索深度（默认 64）
   - 值越大精度越高，但速度越慢
3. **距离度量**：
   - 余弦相似度：cos(θ) = (A·B) / (||A|| × ||B||)
   - 欧氏距离：L2 = √(Σ(Ai - Bi)²)
   - 内积：IP = A·B（适用于归一化向量）

**性能优化**：
- 批量插入：每 100-1000 个文档批量插入一次
- 分区管理：按 ragNamespace 分区，支持命名空间隔离
- 缓存策略：热点查询结果缓存到 Redis
- 索引压缩：使用 IVF_PQ 量化压缩，减少内存占用

**扩展性**：
- 支持水平扩展（多节点分布式部署）
- 支持十亿级向量检索
- 毫秒级响应时间（百万级向量库）

**依赖服务**：
- Spring AI VectorStore
- Milvus 2.3.3+

## 📊 功能对比：向量检索 vs BM25 检索

| 对比项 | 向量检索 | BM25 检索 |
|--------|---------|----------|
| **原理** | 向量空间最近邻 | 词频统计匹配 |
| **匹配方式** | 语义相似 | 关键词精确 |
| **理解能力** | ✅ 理解语义 | ❌ 只看字面 |
| **速度** | 较慢（需要向量计算） | 快（纯统计） |
| **同义词** | ✅ 支持 | ❌ 不支持 |
| **新词/罕见词** | ✅ 可能理解 | ✅ 效果好 |
| **实现依赖** | 嵌入模型（OpenAI、BGE 等） | Lucene/Elasticsearch |

### 实际案例

**查询：「如何学习编程」**

向量检索结果：
- 「新手入门 Python 编程指南」✅（语义相似）
- 「从零开始学 Java 开发」✅（语义相似）
- 「编程学习路线推荐」✅（语义相似）

BM25 检索结果：
- 「如何学习编程」✅（精确匹配）
- 「学习编程的方法」✅（部分匹配）
- 「编程教学视频」✅（部分匹配）

**混合检索 = 兼得两者的优点**：
- 向量检索找到语义相关的内容
- BM25 找到关键词精确匹配的内容
- RRF 融合两个结果，互补长短

## 🎯 完整工作流程

### 文档添加流程

```
1. 用户上传文件（PDF/Word/TXT/Markdown）
         ↓
2. DocumentParserService 解析文件内容
         ↓
3. DocumentChunkingService 智能分块
   - 识别标题层级
   - 按段落合并
   - 计算 token 数
   - 添加重叠部分
         ↓
4. 构建 Document 对象
   - ID: UUID
   - Content: 分块文本
   - Metadata: documentId, chunkIndex, headingPath, ragNamespace
         ↓
5. 多存储引擎并行存储
   - VectorStore → Milvus（向量）
   - HybridRetrievalService → BM25（稀疏索引）
   - RagChunkMapper → PostgreSQL（原文）
         ↓
6. 返回结果
   - documentId
   - chunkCount
   - totalTokens
```

### 知识检索流程

```
1. 用户查询：「什么是 AI Agent？」
         ↓
2. 查询优化层
   - QueryRewriter: 重写为「AI Agent 的定义和概念」
   - HyDE: 生成假设文档
   - MQE: 扩展到 ["AI Agent 是什么", "AI Agent 的特点", ...]
         ↓
3. 混合检索层
   - 向量检索：Milvus 搜索语义相似的向量
   - BM25 检索：关键词匹配
   - RRF 融合：合并两个结果集
         ↓
4. 后处理层
   - 语义去重：去除相似度>0.85 的重复内容
   - MMR 重排序：平衡相关性和多样性
   - 上下文扩展：包含相邻分块
         ↓
5. 构建 RAG Prompt
   ```
   基于以下信息回答问题：
   
   [相关知识 1]
   AI Agent 是一种...
   
   [相关知识 2]
   它具有以下特点...
   
   问题：什么是 AI Agent？
   ```
         ↓
6. LLM 生成回答
         ↓
7. 返回答案 + 引用来源
```

## ⚙️ 配置管理

所有优化策略支持动态配置：

```yaml
rag:
  retrieval:
    hybrid:
      enabled: true          # 混合检索开关
    rrf:
      k: 60                  # RRF 参数
  post-processing:
    deduplication:
      enabled: true          # 语义去重开关
      threshold: 0.85        # 相似度阈值
    mmr:
      enabled: true          # MMR 开关
      lambda: 0.7            # 相关性/多样性权重
    context-expansion:
      enabled: true          # 上下文扩展开关
  evaluation:
    enabled: true            # 评估开关
  query-optimization:
    rewrite:
      enabled: true          # 查询重写开关
    hyde:
      enabled: true          # HyDE 开关
    mqe:
      enabled: true          # MQE 开关
```

## 🔍 评估体系

### RAGAS 评估框架

- **忠实度**（Faithfulness）：回答是否基于检索内容
- **答案相关性**（Answer Relevance）：回答是否切题
- **上下文相关性**（Context Relevance）：检索内容是否有用
- **上下文召回率**（Context Recall）：是否找到所有相关文档

### A/B 测试

- 对比不同检索策略的效果
- 对比不同重排序参数的影响
- 对比有无查询优化的差异

### 监控仪表盘

- 检索请求总数
- 平均响应时间
- 检索结果数量分布
